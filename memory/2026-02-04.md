### MEV Capital Analysis (Key Source)
Read deep analysis from MEV Capital on Symbiotic slashing risks:
- VetoSlasher 3-phase lifecycle: requestSlash â†’ vetoSlash â†’ executeSlash
- Resolver responsiveness is the #1 risk (unresponsive resolver = slash executes)
- Middleware governance is critical (who controls slash() determines everything)
- `resolverSetEpochsDelay >= 3` specifically prevents resolver removal during pending slashes
- The delay ensures resolver ALWAYS outlasts the slash period (epochDuration)

### VetoSlasher Timing Analysis (My Conclusions)
Traced multiple attack scenarios:
1. **Resolver removal during pending slash:** SAFE. resolverSetEpochsDelay >= 3 means the change takes effect 2+ epochs AFTER slash period expires
2. **First resolver set bypass veto:** By design. First set is immediate, but slash requests made before resolver set have no veto anyway
3. **latestSlashedCaptureTimestamp ordering attack:** Network controls both slashes, so self-harm only. No profit motive.
4. **Malicious operator self-slashing via BurnerRouter:** By design. Stakers should verify burner config before depositing.

### Strategy Pivot
VetoSlasher timing edge cases are largely exhausted. The interlocking constraints (vetoDuration < epochDuration, resolverSetEpochsDelay >= 3, captureTimestamp bounds) work together to prevent abuse. Certora did FORMAL VERIFICATION on slashing. Finding something here means finding what formal verification missed.

**New strategy: Cross-contract interaction bugs during slashing.** The slash lifecycle touches Vault + Slasher + Delegator + Hook + Burner in ONE transaction. Formal verification likely checked each contract in isolation, but COMPOSITION is where bugs hide. This is our best shot at High/Critical.
